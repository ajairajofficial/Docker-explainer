How Docker Really Works — and How to Actually Port Your Containers

Author — Ajai RajA product of iolinked.com

By the end of this you'll understand what a container is at the kernel level, why "moving a container" is the wrong mental model, and the exact, repeatable way to lift a running service onto another host without losing your data.

The whole idea in one picture: the image is the blueprint, the container is a live process wearing it, the volume is where your actual data lives. Two of the three move between machines. The middle one you throw away and regrow.

Why this matters

You've got a service running on one box — a database, an automation engine, a reverse proxy — and you need it on another box. Maybe you're outgrowing a VPS. Maybe you're splitting workloads. Maybe the cheap host is being decommissioned next Tuesday.

The naïve instinct is "copy the container over." People go looking for a docker move command. It doesn't exist, and the reason it doesn't exist is the whole point of understanding Docker. A running container isn't a file you can copy. It's a process — alive, pinned to one kernel, holding open sockets and memory. You can't FedEx a running process any more than you can mail a lit candle.

What you can move is the recipe (the image) and the ingredients you care about (the data in volumes). Get that distinction right and migration becomes a five-minute checklist. Get it wrong and you'll proudly stand up a "migrated" database on the new host — empty, because the rows were never in the image to begin with.

So let's build the whole thing from the ground up, then do a real migration end to end.

First principles: what a container actually is

Forget everything marketing-shaped you've heard. Here's the ground truth, built up one piece at a time.

A normal program on Linux is a process — an instance of running code, given a number (its PID, process ID) by the kernel, allowed to see the filesystem, the network, every other process, and as much CPU and memory as it can grab. The kernel is the core of the operating system: the single piece of software that talks to the hardware and hands out resources to everyone.

A container is that same thing — an ordinary Linux process — with three modifications bolted on by the kernel:

  1. Namespaces change what the process can see.
  2. Control groups (cgroups) change what the process can use.
  3. A union filesystem changes what files it sees as its root directory.

That's it. No hypervisor. No guest operating system. No virtual hardware. The container shares the exact same kernel as the host and every other container on the box. When you run a container, you are running a process on your machine that has been lied to about its surroundings and put on a resource budget.

🔓 Unlock — A container is not a tiny VM. A virtual machine boots a whole second operating system on emulated hardware. A container is one process on your real OS wearing a costume the kernel sews for it. That's why a container starts in milliseconds and a VM takes 30 seconds — there's no OS to boot. There's nothing to boot. It's already running; the kernel just fenced it off.

Namespaces — the walls (what it can see)

A namespace is a kernel feature that gives a process its own private copy of some global resource. There are several kinds, and each one hides a different slice of reality:

  • PID namespace — the process sees its own process numbering. Inside, your app is PID 1, the lord of its own little universe. It can't even see the host's other processes, let alone signal them.
  • Mount namespace — its own filesystem tree. The container's / is not the host's /.
  • Network namespace — its own network interfaces, IP, routing table, ports. This is why two containers can both "listen on port 5432" without colliding.
  • UTS namespace — its own hostname.
  • IPC namespace — its own inter-process communication (shared memory, semaphores).
  • User namespace — its own user IDs, so "root inside the container" can map to an unprivileged user on the host.

The analogy: namespaces are one-way glass walls. The process inside genuinely believes it's alone in the building. It looks around and sees its own processes, its own network card, its own filesystem. The walls don't add anything — they subtract the rest of the world from view.

Cgroups — the meter (what it can use)

Walls stop a process from seeing the neighbours. They do nothing to stop it from eating all the food. That's the job of control groups, or cgroups: a kernel feature that caps and accounts for a process group's resource consumption — CPU time, memory, disk I/O, network bandwidth.

--memory=256m on a docker run isn't a suggestion. It's a cgroup limit. Cross it and the kernel's OOM (out-of-memory) killer reaps the process. This is the difference between isolation and politeness: namespaces are isolation, cgroups are the enforced budget that stops one noisy container from starving the host.

The analogy: cgroups are the electricity meter with a hard breaker. You can run whatever you like in your apartment, but pull more than your allotted amps and the breaker trips. No negotiation.

Union filesystem — the layered floor (what files it sees)

A container needs a root filesystem — a /, a /bin, a /etc. Where does it come from? Not a fresh copy per container; that would be wasteful. It comes from the image, assembled by a union filesystem (Docker uses OverlayFS, via the overlay2 storage driver).

A union filesystem stacks several read-only directories and presents them as one merged directory. Each directory is a layer. The bottom layer might be a minimal Debian. The next adds your libraries. The next your app code. Stack them, merge them top-down, and you get a complete filesystem — without ever duplicating the shared lower layers.

When the container writes a file, it doesn't touch the read-only layers. The change goes into a thin writable layer that exists only for that one running container. We'll come back to this — it's the mechanism that makes "your data isn't in the image" literally true.

The container in the middle is a single process. The left box is what the kernel lets it see; the right box is what the kernel lets it use; the strip at the bottom is the filesystem it gets handed. Remove all three wrappers and you're left with postgres running on the host like any other program.

How the layers actually get run: the daemon chain

When you type docker run, a relay race kicks off. Worth knowing, because when something hangs you'll want to know which runner dropped the baton.

The CLI is a thin client. dockerd is the brain. containerd runs the lifecycle, runc does the actual kernel syscalls to build the sandbox, then steps aside. Knowing this chain tells you where to look when a container won't start versus when the daemon itself is wedged.

The unlock that makes porting make sense

Here's the misconception that wrecks migrations, stated plainly so you never fall for it again:

🔓 Unlock — The data isn't in the image. It's in the writable layer or a volume. Everything your container writes after it starts — database rows, uploaded files, logs, generated config — goes into the thin writable layer or into a mounted volume, never back into the read-only image. Pull the same postgres:16 image on a fresh machine and you get an empty database. The image is the install media, not the saved game.

This is by design, and it's the single most important fact for porting. The image is reproducible and cheap to move — it's already on a registry, or you can rebuild it from a Dockerfile. The data is precious, unique, and Docker will not move it for you unless you explicitly tell it to.

That's why best practice is to keep important data in a named volume rather than letting it accumulate in the disposable writable layer. A volume is a directory Docker manages (under /var/lib/docker/volumes/ on the host) and mounts into the container. It outlives the container. Delete and recreate the container a hundred times; the named volume stays. That persistence is exactly what you back up and carry to the new host.

Copy-on-write means writes never mutate the shared image — they're copied into the container's private top layer. Good for speed and disk use. Dangerous if you assume that top layer is durable. It isn't. That's the whole argument for volumes.

How porting actually works — the three levels

What it is (the user's view)

"Porting a container to a new host" means: get the same service running on machine B that's running on machine A, with the data intact. Three pieces travel:

  1. The image — almost always the easy part.
  2. The volumes — the data; the part that actually matters.
  3. The run configuration — your docker-compose.yml, environment variables, port mappings, network setup.

How it works (the mechanism)

Moving the image. You have three honest options, in order of preference:

  • Rebuild from source. If your image comes from a Dockerfile and a docker-compose.yml checked into git, you don't move the image at all — you git pull on host B and rebuild. Reproducible, version-controlled, no binary blobs flying around.
  • Pull from a registry. A registry is a server that stores images (Docker Hub, GitHub Container Registry, a private one). docker push on A, docker pull on B. Clean, and it's how production usually works.
  • docker save / docker load. For air-gapped or offline moves. docker save writes the image — all its layers, history, and metadata — to a tar archive. Copy the tar over, docker load reconstructs it. This is a faithful copy of the image.

Moving the data. This is the part Docker won't do automatically. The standard, battle-tested move: spin a throwaway helper container that mounts the named volume, tar its contents to an archive, ship the archive to host B over SSH, then reverse the process into a fresh named volume. The Docker docs themselves describe exactly this backup/restore pattern. We'll trace it in a moment.

Moving the config. Copy your docker-compose.yml and .env. That's it. (Keep secrets in .env, never baked into the image.)

⚠️ Warning — do not just copy /var/lib/docker. It's tempting to tar the entire Docker data directory and drop it on the new host. It's a footgun. You'll hit storage-driver mismatches, Docker-version incompatibilities, broken layer references, and ownership/permission corruption. The supported, predictable unit of migration is image + named volume + compose file — not the daemon's private internal directory. People who copy /var/lib/docker are the ones posting "my migrated volume won't mount" three days later.

What breaks it (the edge cases experts care about)

This is where shallow tutorials stop and real migrations live or die.

  • Database file-copy across versions. A file-level tar of a Postgres or MySQL volume is fast — but the on-disk format is tied to the database's major version and sometimes the CPU architecture. Tar a Postgres 15 data directory onto a host running the Postgres 16 image and it may refuse to start. For databases, a logical dump (pg_dump / mysqldump) is version-portable and arch-portable, at the cost of speed and a brief lock. Pick file-copy for identical engines on identical arches; pick logical dump when anything differs or when you're not certain.
  • Hot data is inconsistent data. Tarring a volume while the database is writing gives you a torn snapshot — half-written pages, an inconsistent state. Quiesce first: stop the container (or use the engine's snapshot/backup tooling) before you archive. A backup taken mid-write is a backup you can't trust.
  • save vs export confusion corrupts your move. They sound identical and do opposite things. More on this below — get it wrong and you'll lose your entrypoint and your layer history.
  • Architecture mismatch. An image built for linux/amd64 won't run on an arm64 host (a Raspberry Pi, an Ampere VPS, Apple Silicon). Rebuild for the target arch, or use multi-arch images. save/load faithfully preserves the wrong architecture if that's what you saved.
  • Bind mounts vs named volumes. A bind mount maps a specific host path (/home/me/data:/data) into the container. It depends on that path existing on the host. Move to a host without it and the mount is empty or fails. Named volumes are Docker-managed and self-contained — far easier to migrate. Prefer them for anything you'll move.
  • Anonymous volumes vanish. Volumes created without a name are garbage-collected when you docker compose down -v or run docker volume prune. If your data ended up in an anonymous volume, a routine cleanup can silently destroy it before you ever migrate.

🔓 Unlock — docker save and docker export are opposites, not synonyms. save operates on an image and preserves every layer, the build history, and the metadata (your CMD, ENTRYPOINT, ENV). export operates on a container and flattens its filesystem into a single tarball with no history, no layers, and no entrypoint — you get the files, but the image forgets how to run itself. Reconstruct with load (for save) or import (for export). Mix them up and your "ported" image launches into nothing.

Read it left to right: save is for images and keeps everything; export flattens a container into bare files and throws away its identity; commit snapshots a running container into a new image. Crucially, the last column — not one of them carries your volume data.

A worked example: moving a live Postgres from host-a to host-b

Real service, real image, real commands. We've got postgres:16 running on host-a via Compose, persisting to a named volume called pgdata. We want it on host-b, rows intact. Build it in small, verifiable steps — confirm each before the next.

Here's the docker-compose.yml in play on host-a:

services:
  db:
    image: postgres:16
    restart: unless-stopped
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: appdb
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"

volumes:
  pgdata:

Step 1 — confirm what you're actually carrying. Find the volume and prove the data is there before you touch anything.

# on host-a
docker compose exec db psql -U app -d appdb -c "SELECT count(*) FROM orders;"
# -> count: 48213    (note this number — it's your migration's pass/fail test)

docker volume inspect pgdata
# confirms the volume exists and where it lives on the host

Step 2 — quiesce, so the snapshot is consistent. Stop the database before archiving. A backup taken while Postgres is writing is a torn backup.

# on host-a
docker compose stop db

Step 3 — tar the named volume with a throwaway helper. The db container is stopped, so we mount the volume directly by name into a tiny alpine container, archive it, and write the archive to the current directory.

# on host-a, in a working folder
docker run --rm \
  -v pgdata:/from:ro \
  -v "$(pwd)":/to \
  alpine tar czf /to/pgdata.tgz -C /from .

sha256sum pgdata.tgz   # record this; you'll verify it on the far side

What each flag does, since copy-paste teaches nothing: --rm deletes the helper when it exits; -v pgdata:/from:ro mounts your real data volume read-only at /from; -v "$(pwd)":/to mounts your current host folder at /to so the archive lands on the host, not inside the disposable container; tar czf … -C /from . creates a gzipped archive of everything in /from.

Step 4 — ship the artifacts to host-b, over SSH. Three files: the data archive, the compose file, the env file. Encrypted transport, not HTTP.

# on host-a
scp pgdata.tgz docker-compose.yml .env user@host-b:/srv/appdb/

Step 5 — on host-b, restore the data into a fresh named volume. Create the volume, verify the archive survived the trip, then unpack into it — again with a throwaway helper.

# on host-b
cd /srv/appdb
sha256sum pgdata.tgz          # must match the hash from step 3

docker volume create pgdata

docker run --rm \
  -v pgdata:/to \
  -v "$(pwd)":/from:ro \
  alpine sh -c "cd /to && tar xzf /from/pgdata.tgz"

Step 6 — get the image and start the service. The image is the easy half — postgres:16 pulls straight from the registry on up. No save/load needed because it's a public image.

# on host-b
docker compose up -d

Step 7 — verify against your step-1 number. This is the only test that matters.

# on host-b
docker compose exec db psql -U app -d appdb -c "SELECT count(*) FROM orders;"
# -> count: 48213   ✅  same number. The rows came across. Migration confirmed.

If that count matches, you're done — point your app's DNS or connection string at host-b and decommission host-a. If it doesn't, you have a torn snapshot (you skipped step 2), a version mismatch (host-b isn't running postgres:16), or an extraction-path error — check those three in order.

📝 Note — the version-safe alternative. For a major-version jump or an architecture change, swap steps 2–5 for a logical dump: docker compose exec db pg_dump -U app appdb > appdb.sql on host-a, then psql … < appdb.sql into a fresh container on host-b. Slower, locks briefly, but immune to on-disk format differences. When in doubt about versions, dump.

The mental model, recapped

Picture it as moving house. The image is the IKEA catalogue — anyone with the same catalogue assembles identical furniture, so you don't ship furniture, you ship the catalogue page (or just re-download it on arrival). The container is the assembled room with you living in it — you can't box up a room mid-dinner and mail it, so you tear it down here and rebuild it there. The volume is the box of photo albums and hard drives — irreplaceable, unique to you, and the one thing you personally carry, carefully, by hand.

Move the catalogue, recreate the room, guard the box of albums. That's the entire discipline of container migration.

Exercises

  1. Watch the costume go on. Run docker run -d --name demo --memory=128m --cpus="0.5" nginx, then docker top demo and docker inspect demo. Find the namespace IDs under /proc/<pid>/ns/ on the host and confirm they differ from your shell's. You're looking at the walls.
  2. Prove the writable layer is ephemeral. Run a container, docker exec in, create /tmp/iwashere. docker commit it to a new image, then docker run that new image and confirm the file is there — but start the original image fresh and confirm it isn't. You've just watched data live in the layer, not the image.
  3. Break it on purpose. docker export a running container to a tar, docker import it back, then try to docker run the imported image with no command. Watch it fail for lack of an entrypoint. Now do the same with save/load and watch it run. Feel the difference the unlock described.
  4. Migrate a real volume locally. Stand up the postgres:16 Compose file above, insert a few rows, then follow the worked example to tar pgdata, recreate it under a new volume name, and start a second stack pointing at it. Verify your row count survives.
  5. Make a database survive a version jump. Take that same Postgres data and move it via pg_dump/psql into a postgres:17 container. Then try the raw file-copy method into postgres:17 and document what happens. Now you'll never guess which method to use again.
  6. Find your anonymous volumes. Run docker volume ls -f dangling=true on a host you've used for a while. Anything important in there is one docker volume prune away from gone. Decide what should be a named volume instead.

Cheat sheet

WHAT A CONTAINER IS
  one host process + namespaces (what it sees) + cgroups (what it uses) + overlayfs (its root)
  shares the host kernel — no guest OS, no hypervisor

WHAT PERSISTS WHERE
  image            read-only layers, reproducible, portable
  writable layer   per-container, ephemeral, DIES on rm — don't keep data here
  named volume     Docker-managed, outlives the container — keep data HERE

MOVE THE IMAGE (pick one)
  rebuild     git pull + docker compose build           # best: reproducible
  registry    docker push  /  docker pull               # production default
  offline     docker save img -o img.tar  ->  docker load -i img.tar

MOVE THE DATA (Docker won't do it for you)
  backup    docker run --rm -v VOL:/from:ro -v "$PWD":/to alpine tar czf /to/VOL.tgz -C /from .
  ship      scp VOL.tgz user@host-b:/path/        # + compose.yml + .env
  restore   docker volume create VOL
            docker run --rm -v VOL:/to -v "$PWD":/from:ro alpine sh -c "cd /to && tar xzf /from/VOL.tgz"
  databases prefer pg_dump / mysqldump across version or arch changes

save  vs  export        save = image (layers+history+CMD) | export = flat container fs (no history)
load  vs  import        load  reverses save              | import reverses export

NEVER
  copy /var/lib/docker between hosts        # storage-driver / version / ownership footguns
  tar a volume while the DB is writing       # torn snapshot — stop the container first
  bake secrets into the image                # keep them in .env / secrets

Glossary

  • Kernel — the core of the OS that talks to hardware and allocates resources. Containers share the host's kernel.
  • Process — a running instance of a program, identified by a PID. A container is a process with restrictions applied.
  • Namespace — a kernel feature giving a process a private view of a global resource (processes, network, mounts, hostname, IPC, users). Controls what a container can see.
  • cgroup (control group) — a kernel feature that caps and accounts for resource use (CPU, memory, I/O). Controls what a container can use.
  • Union filesystem / OverlayFS — stacks read-only directories plus one writable directory into a single merged filesystem. How an image becomes a container's root.
  • Layer — one read-only slice of an image, produced by one build step. Layers are shared across images and containers to save space.
  • Copy-on-write — modifying a file from a read-only layer copies it into the writable layer first; the original is never touched.
  • Image — a stack of read-only layers plus metadata (its CMD, ENV, etc.). The reproducible, portable blueprint for containers.
  • Container — a running (or stopped) instance of an image: the host process with its namespaces, cgroups, and overlay mount.
  • Writable layer — the thin read-write layer on top of a running container. Ephemeral; destroyed when the container is removed.
  • Volume — host storage managed by Docker and mounted into a container; persists independently of the container. Named volume has an explicit name; anonymous volume is auto-named and easily pruned.
  • Bind mount — a specific host path mounted into a container; depends on that path existing on the host.
  • Registry — a server that stores and distributes images (Docker Hub, GHCR, private registries) via push/pull.
  • dockerd / containerd / runc — the daemon, the runtime, and the low-level tool that respectively expose the API, manage container lifecycle, and make the kernel syscalls that build the sandbox.
  • OCI (Open Container Initiative) — the standards body whose image, runtime, and distribution specs let images built by Docker run under other runtimes like Podman.
